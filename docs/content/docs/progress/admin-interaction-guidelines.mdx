# 运维后台交互规范

本规范约束 `/admin` 下的运营与系统管理页面。技术栈保持 React、Ant Design 和现有主题体系；统一的是信息层级与操作节奏，不复制其他产品的视觉皮肤。

## 列表页结构

每个列表页按以下顺序组织：

1. `AdminPageHeader`：左侧为标题与一句用途说明，页面级主操作放右上角。
2. `ListToolbar`：搜索、状态/类型筛选、已生效筛选 chip、重置和列设置；次要操作放工具条右侧。
3. `TableSurface`：承载桌面表格；首次加载使用不少于 8 行的 `AdminTableSkeleton`，保留已有数据的刷新只显示分区 loading。
4. 分页：属于表格底栏并吸底；服务端列表统一支持 `page`、`pageSize` 和真实总数。

筛选、页码和每页数量写入 URL。关键词使用 `filter`，常用分面使用 `status`、`role` 等稳定键；默认值不写入 URL。切换筛选时页码回到 1，刷新、分享和浏览器前进后退必须可复现当前列表。

## 操作列

- 操作列最多显示一个主操作和一个 `AdminRowActions` 更多菜单。
- 查看详情可作为主操作；编辑、启用/停用、删除和低频操作进入更多菜单。
- 列表内只展示状态 `Tag`，禁止使用会立即写入的 `Switch` 或角色 `Select`。
- 停用和删除必须说明影响范围并二次确认；角色变更必须进入编辑抽屉确认。
- 日志列表只展示定位问题所需摘要，`path`、`upstreamUrl` 和错误全文进入详情抽屉。

## 空态与加载态

- 无数据：标题使用“暂无数据”或具体对象文案，说明数据如何产生；存在创建能力时可提供 CTA。
- 筛选无结果：统一使用“没有符合筛选条件的数据”，说明调整搜索词或筛选条件。
- 首次读取：使用 `AdminTableSkeleton`，至少 8 行；不得用全局“正在更新数据”覆盖整个后台。
- 后续读取：保留上一次数据，只在当前表格分区展示 loading，避免切换条件时页面跳空。

## 表单形态

- 创建和编辑长表单使用右侧 `Drawer`；系统配置使用全页 `SettingsSectionCard`。
- `Modal` 仅用于确认和调账备注等短输入，不承载长配置表单。
- 抽屉表单关闭前检查脏状态；有未保存修改时提供“放弃修改 / 继续编辑”确认。
- 保存、启用、权限、角色、停用和删除属于强校验写路径，接口失败时保留表单并明确提示。
- 已保存的密码或密钥统一显示“已配置 · 留空不改”，接口不回显明文，留空保持原值。

## 设置卡片

登录注册、邮件、OSS 等全页配置统一使用 `SettingsSectionCard`：头部包含图标、标题、说明与状态 Tag；中部承载表单；底部保存条包含安全/更新时间说明和唯一主保存按钮。不要在卡片内部再嵌套装饰卡片。

## 响应式与可访问性

- 桌面表格允许横向滚动；移动端逐步替换为摘要卡片，并把筛选收进按钮触发的浮层。
- 图标按钮必须有可访问名称，更多菜单和抽屉支持键盘操作。
- 动效只表达状态与空间关系，并遵循 `prefers-reduced-motion`。

## 当前落地与后续

用户管理是标杆页：已拆为 panel、columns 和 drawer，具备 URL 状态、列显隐、筛选 chip、骨架、双语义空态、详情/编辑抽屉、脏检查、统一行操作和批量停用。后台已切换为 `/admin/users`、`/admin/channels`、`/admin/logs` 和 `/admin/settings/*` 真路由，渠道、日志、分镜提示词和存储配置分别维护；后续列表继续按同一模板迁移移动端摘要和列显隐能力。
